Project OverviewNimble (thenimble.in) is a publishing platform for writers and readers. The existing stack runs a React frontend connected directly to Supabase, which provides the database, authentication, file storage, and auto-generated APIs in one managed service.As traffic grows, this setup has hit its limits. Because the React frontend calls Supabase directly, every page load by every reader hits Supabase's API endpoints with no buffer, no caching, and no control. The result is rate limiting that makes the site unstable.Core problemThere is no backend layer between the frontend and the database. Every reader request goes directly to Supabase. When traffic spikes, the free-tier rate limits are hit and the site becomes unavailable.1.1  What needs to changeThe migration introduces a dedicated Node.js backend that sits between the React frontend and all data services. This backend is the only layer that talks to the database, handles authentication, processes image uploads, and manages caching. The React frontend stays — only the backend changes.New flowReact frontend  →  Node.js + Fastify API  →  Neon PostgreSQL  |  Cloudflare R2  |  Upstash Redis  |  Clerk Auth1.2  Why this fixes the problemThe Node.js backend pools database connections — Neon is never hammered by raw frontend requestsUpstash Redis caches popular articles — the database is not queried on every page loadRate limiting is enforced at the API level with full control over thresholdsCloudflare R2 stores images with no rate limits on reads and zero egress feesThe entire stack runs on free tiers — no monthly cost at current traffic levels2.
